Last updated: 2026-05-25

This is the single task source of truth for Yale Research. Keep it operational: the top sections should say what to do next, while the lower sections preserve stable context, completed milestones, and verification commands.

The roadmap follows the hard-pivot decision: keep runtime centered on canonical `ResearchEntity` infrastructure, finish Beta seeding and search confidence, then move deliberately toward production scraper rollout and post-Beta cleanup.

### Freshness And Discovery Direction

Freshness should be domain-specific rather than one global "refresh the database" job. Faculty identity, lab existence, departments, centers, and official profiles are slow-changing and can be refreshed around academic cycles, roughly before fall, before spring, and after the school year. Programs, fellowships, deadlines, posted opportunities, application links, and scholarly activity change faster and should get the most automation.

Do not assume Render can fetch every Yale source. Some Yale directory, profile, roster, YSM/YSE, and department pages may block Render IPs and work only from Yale VPN. Keep Meilisearch private to Render: local VPN jobs may fetch Yale-blocked sources and write MongoDB, then mark search sync as pending. A Render-side job should read MongoDB, sync or swap-rebuild the prefixed Meilisearch indexes, and mark the refresh complete.

Deployment planning should preserve the existing branch/service split: `beta` deploys to the Beta Render service and `main` deploys to the paid production Render service. Beta and production may share Meilisearch infrastructure only through separate `MEILISEARCH_INDEX_PREFIX` values (`beta` on Beta Render, `prod` on production Render). Local development is intentionally different from Render and may use Docker Meilisearch with an empty/local/beta prefix for testing. Future CI/CD or deploy questions should treat the missing piece as environment validation and deploy gating, not adding prefix support from scratch: deployed Beta must not run with `MEILISEARCH_INDEX_PREFIX=prod`, deployed production must not run with `MEILISEARCH_INDEX_PREFIX=beta`, and local/dev/test should remain permissive unless an explicit production write command is being run.

Treat freshness, discovery, and promotion as separate workflows:

- **Freshness:** revisit known records and known source pages using cheap probes, hashes, source counts, deadline windows, and current-state audits before running expensive scrapers.
- **Discovery:** find new programs, fellowships, postings, join pages, opportunity pages, and scholarly updates through shallow source crawling, link scoring, public APIs, and review queues. Freshness probes alone do not improve coverage.
- **Promotion:** validate source behavior and materializer changes in Beta, then run production-ready sources directly against production with backup/rollback, gates, and Meili sync.

Near-term automation should therefore be: Render cron for public-safe refreshes, pending Meili sync, source/search health checks, and operator-board updates; local VPN refresh packs for Yale-blocked canonical sources; manual reviewed batches for expensive or access-claim-producing sources such as broad LLM, broad OpenAlex, and arXiv. The operator board should show stale sources, pending VPN jobs, pending Meili sync, weak search queries, discovery candidates, source failures, and the exact command to run when a human/VPN is required.

Foreseeable risks to keep visible while designing this:

- **Beta/production drift:** Beta may look good because of manual cleanup while production later runs only the automated path. Track source version, materializer version, run IDs, and search sync status per environment so the operator board can compare posture.
- **Pending Meili sync debt:** Local VPN jobs can make MongoDB correct while production search stays stale. Treat pending production Meili sync older than a short window as an operator-board warning.
- **Discovery backlog:** Crawlers can find candidate pages faster than they are reviewed. Rank candidates by student impact, deadline urgency, weak-query coverage, and entities with no ways-in instead of creating an unbounded queue.
- **False opportunity creation:** Generic "join us" or "contact us" pages must not become posted opportunities. Only active/time-bound postings should create `PostedOpportunity`; weaker pages become access evidence, contact-route candidates, or review items.
- **Deadline freshness:** Programs, fellowships, application links, and posted opportunities are more time-sensitive than faculty identity. They need their own fast lane and should not wait for semester-cycle canonical refresh.
- **Quiet search regression:** More data can make search worse. Keep a curated query review set and check top-result quality after meaningful data or index changes, not just result count.
- **Cost creep:** Public APIs, LLM extraction, broad OpenAlex/arXiv runs, and shallow crawling need WorkPlanner, changed-target-only execution, accepted target lists, hard caps, and source cost classes.
- **Source blocking and partial parses:** Yale or public pages can be blocked or structurally changed while still returning a partial page. Source probes should detect selector health and count drift before materialization.
- **Hidden manual state:** One-off repairs can be overwritten by future refreshes unless represented as source-backed observations, accepted artifacts, durable locks, or explicit review decisions.
- **Premature worker infrastructure:** Do not buy a durable worker system before planning/status proves the need. Start with `RefreshJob`/source readiness/pending sync/discovery-candidate visibility, then add workers only when cron and local packs cannot carry the workflow.

### Post-Promotion Beta And Development Downsizing

After the accepted Beta dataset is copied or otherwise promoted to production, Beta and development should stop carrying full production-scale duplicate data by default. Production becomes the durable full dataset. Beta should become a smaller but representative staging dataset, and development should become smaller still.

Keep Beta robust by preserving coverage slices rather than raw volume:

- representative `student_ready`, `limited_but_safe`, `operator_review`, and `suppressed` records
- examples across schools, departments, entity types, Programs/Fellowships states, access evidence states, contact-route states, posted-opportunity states, and weak-profile cases
- known regression fixtures for search queries, source-backed descriptions, Programs deadlines, duplicate/disambiguation cases, archived canonical targets, and Meili sync behavior
- recent `Source`, `ScrapeRun`, and accepted-artifact metadata needed to test operator-board and gate behavior

Prefer a reproducible downsample/fixture-seed command over hand deletion. The command should be dry-run-first, production-blocked by default, and able to rebuild prefixed Meili indexes afterward. Keep enough observations and run reports to test materializers, freshness, discovery, and review queues, but prune bulky duplicated raw observations and superseded historical data once production has the accepted full copy and backups exist.

Development should use an even smaller deterministic seed for fast local tests: a compact research/search/program corpus, a few admin review queues, and explicit edge cases. It should not require full Beta or production data to test ordinary UI, search, scraper, and materializer behavior.

### Carry-Forward Guardrails

- `ResearchEntity` is canonical at runtime. Remaining `ResearchGroup`, `lab`, and `researchGroupId` names are migration residue unless a file is explicitly migration/rollback support.
- `EntryPathway`, `AccessSignal`, `ContactRoute`, and `PostedOpportunity` stay first-class. Do not turn exploratory outreach, course credit, fellowship funding, or thesis advising into fake posted opportunities.
- Course credit, paid RA work, fellowship funding, and thesis advising are usually formalization after home/mentor fit. Only a hosted program, mentor-matching program, or real posted application should behave as an entry route or posted opportunity.
- Papers and publications are research-activity context through compact `research_scholarly_links` and `research_scholarly_attributions`, not access evidence or a local paper archive. OpenAlex stays an internal discovery pointer unless no better public destination exists.
- Official Yale profile pages are valid profile enrichment sources, but faculty bios/topics must stay separate from verified lab/entity descriptions. PI-profile synthesis may help sparse faculty research areas without implying lab verification or undergraduate availability.
- `lab-microsite-description-llm` remains a reviewed repair tool, not a blind bulk backfill. Accepted descriptions need useful full text, source evidence, quote verification, clean derived short text, and post-chunk integrity/data-quality checks.
- Separate current-state quality from incoming-write quality. A clean current database scorecard does not prove a new source is safe; a clean run report does not prove existing data is trustworthy.
- Admin weak-profile triage is a runtime quality lens for operators. Non-admin student surfaces should stay calm and source-honest with `Limited public description` style language.
- Department coverage should be repaired through canonical aliases and real source coverage. Do not guess unresolved source-unit labels into student-facing department badges.
- GitHub Actions plus Render remain the CI/CD spine. Keep production promotion manual until the Beta gate is consistently boring.
- Better Stack monitoring is a post-deploy setup item, not pre-launch infrastructure. After stable deployed URLs exist, create `Yale Research API` and `Yale Research Web` uptime monitors with email alerts, then add a `Yale Research Scraper` heartbeat only after scraper jobs are scheduled. Keep Yale-specific product/data-quality analytics in the admin dashboard.

## Production Gate

Run this gate before copying Beta to production or enabling production scraper writes:

- Server CI posture: `npx tsc --noEmit -p server/tsconfig.json`, `yarn --cwd server test`, and high-severity dependency audit in CI.
- CI/CD posture: GitHub Actions remains the release spine; Beta may auto-deploy from the Beta branch after required checks pass, while production stays manual via `workflow_dispatch` or Render deploy hook until the gate is consistently low-risk.
- Beta data posture: `yarn --cwd server beta:data-quality --strict --include-samples` exits 0, with any warnings documented or queued.
- Scraper materialization posture: `yarn --cwd server scraper:integrity-gate --include-samples` exits 0 or the exact duplicate/current-state failures are fixed before promotion.
- Semantic Research posture: if `RESEARCH_SEARCH_SEMANTIC=true`, run `RESEARCH_SEARCH_SEMANTIC=true yarn --cwd server beta:readiness --confirm-beta-backup --strict`; the gate blocks unless Meili reports embedded `researchentities` documents.
- Release posture: backup/restore point exists, Meili rebuild/sync plan is ready, and smoke checks cover `/research`, `/programs`, `/opportunities/:id`, retired `/pathways` redirects, retired `/api/listings`, and guarded admin routes.
- Observability posture: after production URLs are stable, configure Better Stack monitors for `Yale Research API` at `/api/health` and `Yale Research Web` at the public root with email alerts; add the scraper heartbeat only once scraper jobs are actually scheduled.

Before the actual production deploy, revisit these value points:

- **Restore drill:** prove an Atlas backup can be restored into a temporary database before relying on it for production promotion.
- **Dataset versioning:** record a human-readable dataset version, source run set, materializer/search build, and Meili index build so post-launch bug reports can be tied to a concrete data snapshot.
- **Promotion diff:** generate a readable Beta-to-production diff before copy or guarded delta apply: created/archived/changed research entities, Programs, visibility tiers, descriptions, source URLs, posted opportunities, contact routes, and search index counts.
- **Search rollback:** treat MongoDB and Meili rollback separately. Use swap rebuilds for production Meili and keep enough previous-index state to recover from a bad search rebuild without restoring MongoDB.
- **Public payload privacy gate:** recheck student-facing payloads for raw scraped emails, phone numbers, forbidden source URLs, and unguarded contact fields. Official routes and policy-filtered contact routes should win over raw scraped contact data.
- **Golden evaluation set:** keep a compact regression set for student search queries, expected good results, bad matches to avoid, Programs cycle states, real posted opportunities, and pages that must not become opportunities.
- **Admin queue ownership:** every operator-board queue should have clear actions such as accept, reject, suppress, defer, needs source, or lock field. Visible queues without decisions are still operational debt.
- **Seed privacy boundary:** decide which Beta/dev seed modes may contain real Yale data and which fixtures must be anonymized for shared development or open-source tests.
- **Minimal alerting:** add alerts or operator-board red states for pending production Meili sync, stale deadline sources, scraper materialization errors, empty/sharply smaller Meili indexes, and sudden drops in public `/research` result count.
- **No-change success:** semester refreshes should report "no meaningful changes" as a valid successful outcome, not as a failed run or a reason to broaden scraping.
